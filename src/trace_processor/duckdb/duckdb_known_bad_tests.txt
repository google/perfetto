# DuckDB lane: KNOWN-BAD diff tests.
#
# Each line is one diff test that DuckDB BINDS AND RUNS (it is NOT rejected by
# the support predicate and does NOT error into the SQLite fallback) but whose
# output DIVERGES from the SQLite golden for a reason that cannot be cleanly
# fixed at the engine/iterator layer without breaking the SQLite lane. These are
# the irreducible divergences left after the deterministic fixes and the "list,
# don't guard" policy shift (USING/double-quote guards relaxed, row-order guard
# relaxed, column-name override, double-quote-literal repair, polymorphic
# extract_arg via UNION, runtime view/table/scalar-function mirroring).
#
# This file is the EVIDENCE LEDGER for the fallback lane: every entry was
# observed by running the full suite with the DuckDB engine ON and fallback ON
# (PERFETTO_ENABLE_DUCKDB=1, no disable-fallback) and confirming the test fails
# only because DuckDB ran the (output-producing) query and its result diverged.
# CRUCIALLY, for the TIE_BREAK majority the DATA is correct - only the row ORDER
# differs - which was verified by checking the expected/actual row SETS match.
# No entry here is a wrong-VALUE bug in the migrated functions.
#
# Categories:
#   TIE_BREAK       Identical row SET, different row ORDER: the query has no
#                   total ORDER BY (or a LIMIT / GROUP BY without one), so the
#                   two engines order/select equal-key rows differently. The
#                   golden encodes SQLite's arbitrary order. Statically
#                   undetectable; irreducible without rewriting the query.
#   FORMATTING      double->text formatting differs. SQLite's CAST(real AS TEXT)
#                   uses ~15 significant digits (%!.15g); DuckDB emits the
#                   shortest round-trippable form. The divergence happens inside
#                   the SQL CAST (cast_string!(real_value)), not at the iterator
#                   (a raw double COLUMN renders via the shared serializer and
#                   matches), so it cannot be normalized at the value layer.
#   NAN_NULL        a NaN real, stringified inside SQL by cast_string!, renders
#                   as "[NULL]" in SQLite but "nan" in DuckDB. (A raw NaN double
#                   COLUMN is already normalized to NULL by the iterator.)
#   MULTI_STATEMENT the multi-statement split path ("Result rows were returned
#                   for multiple queries") or a shell setup query erroring under
#                   the lane; a pre-existing engine split-path issue, independent
#                   of routing.
#   OTHER           divergence of another kind - UNION column-type coercion, a
#                   view-body GROUP BY dialect difference, or an unaliased
#                   expression column inside a WITH/compound query (the
#                   column-name override only fires for a leading SELECT). See
#                   the per-line reason.
#
# NOTE: this is the "runs-but-diverges" set only. The much larger set of tests
# that DuckDB CANNOT run (PerfettoSQL `!` macros, span_join/other SQLite virtual
# tables, C++ table-valued intrinsics like _tree_filter / experimental_flamegraph
# / __intrinsic_stdlib_*, SQLite loose-typing comparisons) correctly FALL BACK to
# SQLite and PASS in the fallback lane; they are not listed here.
#
# FOLLOW-UP: wiring this file into the diff runner as a DuckDB-lane exclusion is
# a separate task; for now it is documentation + an evidence ledger.
